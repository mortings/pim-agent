How it's wired

Each demo is a single self-contained HTML file. The browser only knows a Worker URL and a shared passphrase. Bluestone credentials live as Wrangler secrets on the Worker — the page never sees them.

Browser  ─►  Cloudflare Worker  ─►  Bluestone PIM MCP  ─►  Bluestone PIM
demo.html      pim-agent-proxy        jsonrpc 2.0           tenant
shared-secret  3 Bluestone headers    tools/call

Redstone PIM takes it further: the same single-file app runs fully standalone on local data, and a ?mcp=1 toggle swaps its data layer onto the live tenant through the very same Worker → MCP path — real reads, real completeness scores, and attribute writes (set_product_attribute, create_product, update_product_name) — degrading gracefully wherever the MCP has no matching tool. Same UI, flip a switch, and it's editing your real catalog — which is the honest division of labour: vibe-coding produced the front-of-house in days, while the scaled, multi-language, governed backend it depends on is a multi-year platform you keep and build on, not something you conjure from scratch.
